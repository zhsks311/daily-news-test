### Deloitte의 EKS/vCluster 사례: ephemeral environment를 “제품”처럼 운영하기
- **어떤 문제를 다루는 글인가:** QA 팀이 격리된 테스트 환경을 필요로 할 때마다 전용 EKS cluster, ALB, Route 53, monitoring agent를 새로 준비하면서 30~45분의 provisioning delay와 플랫폼 팀 병목이 생긴 문제를 다룬다.
- **설계 선택지 또는 아키텍처 패턴:** 단일 Amazon EKS host cluster를 기반으로 vCluster를 올리고, 각 QA 환경은 virtual cluster로 격리한다. ingress controller, storage controller, monitoring 같은 공통 platform service는 host cluster에서 공유하고, ALB는 path-based routing으로 여러 virtual cluster의 앱을 한 load balancer 뒤에 둔다.
- **트레이드오프:** 전용 cluster 방식은 isolation은 명확하지만 control plane, load balancer, monitoring agent가 중복되어 비용과 운영 부담이 커진다. vCluster 방식은 비용과 속도에서 유리하지만, host cluster 장애 영향 범위, tenant별 quota, RBAC, network policy를 더 엄격히 설계해야 한다.
- **내 서비스에 적용한다면 체크리스트:**
  - preview/test environment 생성 시간이 5분을 넘는지 측정한다.
  - 환경별로 중복 배포되는 ingress, DNS, monitoring, logging component를 목록화한다.
  - virtual cluster/namespace 격리로 충분한 workload와 반드시 전용 cluster가 필요한 workload를 구분한다.
  - shared ALB나 shared controller 장애가 모든 테스트 환경에 미치는 blast radius를 정의한다.
- **읽고 남길 한 문장 원칙:** 테스트 환경도 production처럼 platform product로 설계하면, 개발 속도와 cloud cost가 동시에 개선된다.
- **출처 URL:** https://aws.amazon.com/blogs/architecture/deloitte-optimizes-eks-environment-provisioning-and-achieves-89-faster-testing-environments-using-amazon-eks-and-vcluster/

### Kubernetes v1.36 suspended Job resource mutation: batch queue가 자원을 재협상하는 패턴
- **어떤 문제를 다루는 글인가:** Job 생성 시점의 자원 요청이 실제 실행 시점의 cluster capacity와 맞지 않아 대기열이 길어지거나 Job을 삭제·재생성해야 했던 문제를 다룬다.
- **설계 선택지 또는 아키텍처 패턴:** Job을 `spec.suspend: true` 상태로 제출하고, queue controller가 실행 직전에 CPU/memory/GPU 요청량을 조정한 뒤 `spec.suspend: false`로 재개한다. 이 방식은 ML training, batch analytics, CronJob의 특정 실행분처럼 “덜 받더라도 실행하는 편이 낫다”는 workload에 맞다.
- **트레이드오프:** resource downsize는 cluster 활용률을 높이고 대기 시간을 줄일 수 있지만, 실행 시간이 길어지고 SLA가 흔들릴 수 있다. GPU 수를 줄이는 경우 학습 코드가 distributed training topology 변경을 견딜 수 있는지도 확인해야 한다.
- **내 서비스에 적용한다면 체크리스트:**
  - Job 재생성 때문에 사라지는 metadata/status/history가 운영상 중요한지 확인한다.
  - queue priority별 최소·최대 CPU/GPU 정책을 문서화한다.
  - “느리게라도 실행” 가능한 batch와 “정해진 자원 없이는 실패 처리”해야 하는 batch를 분리한다.
  - resource 변경 이벤트를 감사 로그와 비용 대시보드에 남긴다.
- **읽고 남길 한 문장 원칙:** 배치 플랫폼은 제출된 요구사항을 그대로 실행하는 시스템이 아니라, 현재 용량과 비즈니스 우선순위에 맞춰 자원을 재협상하는 시스템이어야 한다.
- **출처 URL:** https://kubernetes.io/blog/2026/04/27/kubernetes-v1-36-mutable-pod-resources-for-suspended-jobs/

### Datadog의 Postgres upsert 성능 디버깅: “no-op write”도 WAL 비용을 만든다
- **어떤 문제를 다루는 글인가:** Datadog은 host metadata store에서 비활성 host 정리를 위해 `last_seen` 성격의 upsert를 추가했다가, 값이 바뀌지 않는 upsert인데도 disk write가 2배, WAL sync가 4배 늘어나는 문제를 만났다.
- **설계 선택지 또는 아키텍처 패턴:** 빈번히 갱신되는 field를 기존 wide table에 넣지 않고 별도 table로 분리하고, coarse-grained freshness를 받아들여 update 빈도를 줄인다. Postgres MVCC, HOT update, WAL write amplification을 전제로 쿼리를 다시 설계한다.
- **트레이드오프:** upsert는 애플리케이션 로직을 단순하게 만들지만 conflict row lock과 WAL 기록 때문에 “변경 없는 쓰기”도 비용이 된다. 반대로 update 조건을 더 엄격히 하면 DB write는 줄지만 쿼리 복잡도와 correctness 검증 부담이 늘어난다.
- **내 서비스에 적용한다면 체크리스트:**
  - high-QPS upsert가 실제로 값 변경을 얼마나 만드는지 측정한다.
  - update 대상 column이 index에 포함되어 HOT update를 막고 있지 않은지 확인한다.
  - freshness 요구사항이 초/분 단위인지, 일 단위로 완화 가능한지 결정한다.
  - WAL volume, checkpoint, IOPS, replication lag을 rollout metric으로 둔다.
- **읽고 남길 한 문장 원칙:** 데이터베이스에서 “논리적으로 아무 일도 안 한 쓰기”가 물리적으로도 공짜라고 가정하지 말자.
- **출처 URL:** https://www.datadoghq.com/blog/engineering/debugging-postgres-performance/

### GitHub Enterprise Server 검색 아키텍처 재구축: 검색은 기능이 아니라 resilience dependency다
- **어떤 문제를 다루는 글인가:** GitHub는 GHES 고객을 위해 검색 경험을 더 빠르고 고가용성 있게 만들기 위해 search architecture를 재구축한 과정을 설명한다. 검색은 UI 편의 기능처럼 보이지만, issue, code, audit, 운영 탐색에 연결되어 장애 시 개발 흐름 전체를 막는다.
- **설계 선택지 또는 아키텍처 패턴:** search subsystem을 독립적으로 관찰하고 복구할 수 있게 만들고, indexing path와 query path를 분리해 병목과 장애 전파를 줄이는 방향이 핵심이다. 특히 self-hosted enterprise 환경에서는 고객별 resource envelope이 달라서 default tuning과 운영 가이드가 중요하다.
- **트레이드오프:** 더 분리된 검색 아키텍처는 장애 격리와 scaling에는 좋지만, index consistency, backfill, version upgrade, storage overhead가 복잡해진다. 검색 freshness를 강하게 요구할수록 write path와 indexing cost가 커진다.
- **내 서비스에 적용한다면 체크리스트:**
  - 검색 장애가 핵심 업무 흐름에 미치는 영향을 SLO로 정의한다.
  - indexing lag, failed shard/job, query latency를 별도 지표로 둔다.
  - backfill과 reindex가 운영 중 traffic에 미치는 영향을 테스트한다.
  - read model이 원본 DB와 불일치할 때 사용자에게 어떻게 보일지 정한다.
- **읽고 남길 한 문장 원칙:** 검색 시스템은 “붙이면 좋은 기능”이 아니라, 읽기 모델과 운영 탐색성을 책임지는 별도 제품이다.
- **출처 URL:** https://github.blog/engineering/architecture-optimization/how-we-rebuilt-the-search-architecture-for-high-availability-in-github-enterprise-server/

### GitHub의 eBPF 기반 deployment safety: 배포 도구 자체를 관찰 대상으로 삼기
- **어떤 문제를 다루는 글인가:** GitHub는 deployment tooling에서 circular dependency를 감지·예방하기 위해 eBPF를 활용한 사례를 소개했다. 배포 시스템이 커질수록 “무엇이 무엇을 호출하는지”가 문서보다 runtime behavior에 더 정확히 드러난다.
- **설계 선택지 또는 아키텍처 패턴:** eBPF로 kernel-level event를 관찰해 network/process dependency를 수집하고, 배포 경로의 위험한 순환 참조를 찾아낸다. application instrumentation만으로 보이지 않는 dependency를 runtime telemetry로 보완하는 접근이다.
- **트레이드오프:** eBPF는 낮은 overhead와 깊은 visibility가 장점이지만, kernel version, 권한, 보안 검토, 데이터 volume 관리가 까다롭다. 또한 관찰 결과를 deployment policy로 자동 차단할 때 false positive가 배포 속도를 막을 수 있다.
- **내 서비스에 적용한다면 체크리스트:**
  - 배포 도구, CI runner, service mesh, control plane 간 dependency graph가 최신인지 확인한다.
  - 배포 중 필요한 외부 서비스가 장애날 경우 rollback까지 가능한지 실험한다.
  - eBPF/agent가 수집하는 데이터의 보안·개인정보 범위를 검토한다.
  - 감지 전용, 경고, 차단의 단계별 rollout 정책을 둔다.
- **읽고 남길 한 문장 원칙:** 배포 시스템도 production system이며, 배포 도구의 dependency를 모르면 안전한 배포를 말할 수 없다.
- **출처 URL:** https://github.blog/engineering/infrastructure/how-github-uses-ebpf-to-improve-deployment-safety/

- **우리 개발/운영 플로우 영향:** 장애 대응 agent를 도입하려면 “데모에서 잘 답한다”가 아니라, 과거 incident에서 원인 추론·조치 제안·위험한 행동 억제를 반복 평가해야 한다. agent 운영은 prompt 작성보다 evaluation dataset과 guardrail 운영이 더 중요해진다.
- **도입 전 리스크/비용 포인트:** incident log에는 민감한 고객/인프라 정보가 많아 replay dataset 정제 비용이 크다. 또한 agent가 제안만 하는지, 실제 remediation까지 수행하는지에 따라 승인 체계와 감사 로그가 달라진다.
- **지켜볼 다음 신호:** SRE agent 제품들이 benchmark 숫자보다 incident replay, approval workflow, postmortem integration을 얼마나 제공하는지 보자.
